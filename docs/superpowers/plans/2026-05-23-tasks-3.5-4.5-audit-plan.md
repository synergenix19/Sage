## AUDIT 1 — Notification Pipeline Integrity

This is the highest-priority section. The notification pipeline is the primary safety output of Tasks 4.1–4.5. A silent failure here means clinical flags are never delivered to clinicians.

### 1.1 Method Name: `notify` vs `notify_review_required`

**PRE-POPULATED FINDING — CRITICAL**

`PostgresNotifier` (notification.py:41) defines only `notify_review_required()`. It has no `.notify()` method.

The working tree `output_gate.py:56` calls:
```python
await notifier.notify(session_id=..., user_id=..., source=..., severity=..., reason=...)
```

This call will raise `AttributeError: 'PostgresNotifier' object has no attribute 'notify'` at runtime. The error is silently caught by `except Exception:` at line 63. Net effect: **every clinical and crisis flag silently fails — no clinician notification has ever been delivered in any integration test or manual run of Tasks 4.1–4.5.**

This is the implication: the entire notification pipeline shipped in Tasks 4.1–4.5 is untested at the method boundary. Every test that uses `MagicMock(PostgresNotifier)` is testing a mock object that responds to any method name, not the real class. `MagicMock` auto-creates `.notify()` as a callable attribute on access — so `mock_notifier.notify(...)` succeeds even though the real `PostgresNotifier` has no such method. The only way to catch this class of bug is a test that instantiates the real `PostgresNotifier` against a mocked asyncpg pool (see Check 6.1). Without it, method name drift will recur silently.

**Check to run:**
```bash
# Verify notification.py exposes no .notify() method
grep -n "def notify" src/sage_poc/memory/notification.py
# Expected: only notify_review_required appears

# Verify what output_gate.py currently calls
grep -n "notifier\." src/sage_poc/nodes/output_gate.py
# Expected: must call notify_review_required, NOT notify

# Verify tests are not masking this with MagicMock auto-create
grep -n "mock_notifier_instance\." tests/test_output_gate_clinical_review.py
# Expected: mock attribute name matches actual method name
```

**Fix required before any further build:** Change `output_gate.py:56` from `notifier.notify(...)` to `notifier.notify_review_required(...)` and align the call signature to `(user_id, session_id, reason, source, payload, severity)`. Update tests to mock the correct method name.

### 1.3 Async Task Error Visibility

**MINIMUM BAR ITEM**

**Check:** Does the `asyncio.create_task(_log_clinical_review(...))` call in `output_gate_node` have a done callback for error logging?

```bash
grep -A5 "asyncio.create_task.*_log_clinical_review" src/sage_poc/nodes/output_gate.py
```

Expected: a `.add_done_callback(lambda t: ...)` on the returned task.

Without a done callback, exceptions raised inside the task (including the `AttributeError` from Check 1.1 once the broad `except` is narrowed, or any future exception) are handled by the event loop's default unhandled exception handler — typically a log to `stderr`, not to the application's observability pipeline. In production, a notification failure would be invisible until a clinician reports they never received an alert. This is a silent clinical safety failure.

The fix is three lines, added after `asyncio.create_task(...)`:
```python
_task = asyncio.create_task(_log_clinical_review(...))
_task.add_done_callback(
    lambda t: _log.warning("[output_gate] clinical review failed: %s", t.exception())
    if not t.cancelled() and t.exception() else None
)

## AUDIT 2 — v7 Spec Compliance

### 2.1 Tool Binding: Graph Construction vs Invocation Time (v7 §5.6.2)

**PRE-POPULATED FINDING — ARCHITECTURAL DEVIATION**

v7 §5.6.2: "Tool schemas are bound to the node at graph construction time, not injected into the prompt text."

Actual behavior: `freeflow_respond.py:52` calls `llm.bind_tools(tools)` inside `_invoke_with_tool_loop()`, which is called from `freeflow_respond_node()` on every user turn. This means tool schemas are re-bound on every LLM call, not once at graph construction.

**Check:**
```bash
grep -n "bind_tools\|build_graph\|make_flag_tool\|make_record_tool" src/sage_poc/nodes/freeflow_respond.py
grep -n "freeflow" src/sage_poc/graph.py
```

**Determination to make:**

The deviation is from the **letter** of v7 §5.6.2, not its **intent**. The spec's purpose is to prevent tool schemas from consuming prompt tokens on every turn. LangGraph's `bind_tools()` attaches schemas at the model API level (via the `tools` parameter of the chat completion request), not by injecting schema text into the system or human prompt. Per-invocation binding therefore does not consume prompt tokens — it sends the same JSON schema as query metadata, which is exactly what graph-construction-time binding would do.

The practical reason per-invocation binding is necessary for the POC: tool factories are closures over `user_id` and `session_id` from the live request, which are not available when `build_graph()` is called at server startup. For the Full Build, a tool registry pattern (build tool functions at startup, inject user context at invocation via partial application) would satisfy both the letter and intent of the spec.

- Is this deviation documented? If not, add to Known Deviations section of this file.
- Full Build path: tool registry with `functools.partial(tool_fn, user_id=..., session_id=...)` at invocation, schemas pre-registered at construction.

**Additional edge case — Initial value of `turn_count`:**

### 10.4 Prompt Layer Token Budget with Prior Context Injection

v7 §5.6 budgets approximately 1,100 words across all prompt layers. Layer 5 (prior session context) is estimated at ~100 words. `check_user_history` retrieves up to `top_k=3` session fragments. If each fragment is ~200 words, the total injected prior context is ~600 words — 6x the Layer 5 budget, potentially pushing the total prompt past the 3,000-token zone recommended by the architecture.

**Check:** Is there a cap on the total injected prior context, either at retrieval time or at prompt composition time?

```bash
grep -n "max_context\|truncat\|token\|word.*limit\|[:500]\|[:200]\|\[:100\]" src/sage_poc/nodes/tools/check_user_history.py src/sage_poc/nodes/freeflow_respond.py
```

Expected: one of:
- A word/character cap on each retrieved fragment before formatting
- A total-length cap on the concatenated prior context string
- A `top_k=1` default that limits volume at retrieval time

If none of these exist, the prior context injection has no budget guard. A session with 3 long prior summaries above the 0.6 threshold will silently overflow the prompt budget. Flag as P2 if no cap exists.
