### FAIL / FLAG findings (detailed)

#### FAIL 4.1 — Tool loop exhaustion returns empty string (no fallback)

**Location:** `src/sage_poc/nodes/freeflow_respond.py:72`

```python
    # Safety: exceeded iteration limit — return empty to trigger graceful fallback
    return ""
```

**The problem:** The comment says "trigger graceful fallback" but `freeflow_respond_node` (line 114-121) does not detect an empty return:

```python
    response = await _invoke_with_tool_loop(...)

    return {
        "response_en":   response,   # empty string propagated to output_gate
        ...
    }
```

`output_gate_node` does not check for empty `response_en` either. The result is that after 5 unresolved tool-call iterations, the user receives a blank response. The comment is aspirational — no actual fallback exists at the call site.

**Risk:** Medium. Requires 5 consecutive tool-only responses with no final text, which is unlikely in practice, but produces a silent blank output when it occurs.

**Fix:** In `freeflow_respond_node`, detect empty response and call `resilient_invoke` as fallback:
```python
    if not response:
        response = await resilient_invoke(
            llm, messages[:2], node="freeflow_respond",
            language=state.get("detected_language", "en"), fallback_llm=fallback_llm
        )
```

#### FLAG 2.1 / 8.2 — Invocation-time bind_tools deviation undocumented

**Location:** `src/sage_poc/nodes/freeflow_respond.py:52`

```python
llm_with_tools = llm.bind_tools(tools)
```

**The issue:** v7 §6.5.5 specifies tools are bound at graph construction time. This implementation binds at invocation time (per-turn), injecting `user_id` and `session_id` via closure. This is intentional and correct (closures cannot be serialized into a static graph object), but there is no comment explaining _why_ construction-time binding is not used or citing the v7 deviation.

`check_user_history.py` has an exemplary deviation comment (lines 3-10). `freeflow_respond.py`'s comment at lines 95-96 only explains Sprint 4 import availability — not the binding timing deviation.

**Fix:** Add a comment near line 52:
```python
# Deviation from v7 §6.5.5: tools are bound per-invocation, not at graph construction,
# because user_id and session_id are injected via closure and not known at build time.
llm_with_tools = llm.bind_tools(tools)
```

#### FLAG 6.4 — Empty user_id guard in check_user_history (partial coverage only)

**Location:** `src/sage_poc/nodes/tools/check_user_history.py` — no guard; `src/sage_poc/nodes/freeflow_respond.py:17-18` — guard exists there

**The issue:** `retrieve_prior_context` has no early return for empty/None `user_id`. The guard lives in `_get_prior_context` (the wrapper), so direct calls to `retrieve_prior_context` with empty user_id would attempt a DB query with `user_id=""`. Tests confirm the wrapper path works but do not test direct calls with empty user_id.

**Risk:** Low — the function is only called from `_get_prior_context` in production. But the public signature is misleading.

**Fix (option A):** Add guard to `retrieve_prior_context`: `if not user_id: return ""`
**Fix (option B):** Document in docstring that caller is responsible for user_id validation.

#### FLAG 9.2 — observations JSONB: json.dumps then ::jsonb cast (correct but note-worthy)

**Location:** `src/sage_poc/memory/postgres_repository.py:91`

```python
json.dumps(profile.get("observations", [])),
```

passed as `$12::jsonb`. This is the correct asyncpg pattern for JSONB — asyncpg does not natively serialize Python lists to jsonb without `json.dumps`. The `::jsonb` cast at the SQL layer handles the string-to-jsonb conversion. Not a bug, but unusual enough to warrant a comment for future maintainers who might remove the `json.dumps` assuming asyncpg handles it natively.

#### FAIL 10.4 — Prior context: no length cap on injected fragments

**Location:** `src/sage_poc/nodes/tools/check_user_history.py:32,54-58`

```python
top_k: int = 3,
...
lines = [
    f"{_ATTRIBUTION_PREFIX}: {r['summary_text']}"
    for r in results
    if r.get("similarity", 0) >= _SIMILARITY_THRESHOLD
]
return "\n".join(lines)
```

**The problem:** `summary_text` is injected verbatim. If each session summary is 200-400 words (typical LLM summary output), 3 summaries could inject 600-1200 words into the system prompt. There is no character, word, or token cap on either individual fragments or the total returned string.

**Risk:** Medium. Long prior context can crowd out skill/crisis instructions, cause context-window pressure on short-context models, and inflate costs.

**Fix:** Cap the return value or individual summaries:
```python
_MAX_PRIOR_CONTEXT_CHARS = 800
...
return "\n".join(lines)[:_MAX_PRIOR_CONTEXT_CHARS]
```
Or truncate each fragment: `r['summary_text'][:250]`.
